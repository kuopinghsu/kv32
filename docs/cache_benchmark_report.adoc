This report presents a comprehensive benchmarking study of the I-cache and
D-cache hierarchy of the KV32 RV32IMAC in-order processor.  Thirteen cache
configurations spanning 4 KB to 32 KB total capacity were evaluated across
six benchmarks and three memory back-ends (SRAM, DDR4-1600, DDR4-3200) for a
total of 234 simulation runs.  The key finding is that IC-8K+DC-8K (16 KB
symmetric, 2-way LRU) is the recommended default: it eliminates 90%+ of the
no-cache penalty on DDR4 and provides the best balance of performance, silicon
area, and workload generality.  Configurations below 4K+4K are not
recommended; configurations above 8K+8K yield diminishing returns for most
embedded workloads.

== Introduction

The KV32 is a 32-bit in-order RISC-V (RV32IMAC) processor with a 5-stage
pipeline.  Being in-order, every instruction cache miss causes the fetch stage
to stall until the cache line is refilled from AXI memory -- there is no
out-of-order execution to hide the latency.  The same applies to data-cache
misses in the Memory stage.  This makes cache sizing critically important: the
fraction of cycles spent stalled is directly proportional to the miss rate
multiplied by the refill latency.

This report covers a full sweep of 13 cache configurations across 3 memory
back-ends (SRAM, DDR4-1600, DDR4-3200) and 6 benchmarks (234 runs total, all
PASS).  It serves as a technical guide for selecting cache size and
associativity, explaining why each configuration behaves the way it does, and
providing concrete deployment recommendations.

=== KV32 Pipeline and Cache Architecture

The KV32 pipeline has five logical stages: Fetch -- Decode -- Execute -- Memory -- Write-Back.
Instruction and data caches operate independently but share the same AXI
interconnect to main memory.

Each cache is:

* *Set-associative:* configurable 2-way or 4-way, with LRU replacement
* *Write-back + write-allocate:* a D-cache write miss allocates a line and
  writes back only on eviction
* *Fixed 32-byte cache lines:* each miss triggers an 8-beat AXI burst (32 B /
  4 B per beat)

On a miss:

. The pipeline stalls at the Fetch or Memory stage
. The cache controller issues an AXI read burst to main memory
. On SRAM (1-cycle latency) the burst completes in ~8 cycles
. On DDR4, CAS latency + tRCD add tens of cycles before the first data beat
  arrives, making each miss far more expensive

*Hit rate is not the whole story.* A 95% hit rate at 4 KB can be worse than
a 97% hit rate at 8 KB if the miss penalty is 30 cycles -- the math of
`miss_rate x miss_penalty` drives stall %.  This report shows both.

=== Cache Microarchitecture Details

The KV32 I-cache and D-cache share the same base microarchitecture,
parameterised at synthesis time via three top-level RTL parameters:

[cols="2,1,4",options="header",stripes=even]
|===
| RTL Parameter      | Default | Description
| `CACHE_SIZE`       | 4096 B  | Total cache capacity in bytes; must be a power of two
| `CACHE_LINE_SIZE`  | 32 B    | Fixed; equals one 8-beat AXI burst (32 B @ 4 B/beat)
| `CACHE_WAYS`       | 2       | Set associativity; 2-way or 4-way with LRU replacement
| `FAST_INIT`        | 1       | Reset state machine initialises all valid bits to 0;
                                  cache is available immediately after reset without a
                                  software flush
|===

The number of sets and tag-bit width are derived as:
`NUM_SETS = CACHE_SIZE / (CACHE_LINE_SIZE * CACHE_WAYS)` and
`TAG_BITS = 32 - log~2~(NUM_SETS) - 5` (where 5 = log~2~(32) byte-offset bits).

[cols="3,1,1,1,1",options="header",stripes=even]
|===
| Configuration            | CACHE_SIZE | CACHE_WAYS | NUM_SETS | TAG_BITS
| IC/DC-4K  2-way          |   4096 B   |     2      |    64    |    21
| IC/DC-8K  2-way          |   8192 B   |     2      |   128    |    20
| IC/DC-16K 2-way          |  16384 B   |     2      |   256    |    19
| IC/DC-4K  4-way          |   4096 B   |     4      |    32    |    22
| IC/DC-8K  4-way          |   8192 B   |     4      |    64    |    21
| IC/DC-16K 4-way          |  16384 B   |     4      |   128    |    20
|===

*I-Cache State Machine (read-only, no writeback):*

----
S_IDLE → S_LOOKUP → S_HIT_RD                       (hit: data forwarded next cycle)
                  → S_FILL_AR → S_FILL_R → S_IDLE  (miss: WRAP8 AXI read burst)
----

On a miss the cache controller issues an 8-beat WRAP burst anchored at the
missed word address.  The WRAP burst delivers the critical word first,
minimising the stall window before the pipeline can resume.  Subsequent beats
fill the remainder of the cache line in the same transaction.

*D-Cache State Machine (adds write-back eviction path):*

The D-cache extends the I-cache state machine with two additional states for
dirty-line eviction:

* *Read miss, clean eviction:* `LOOKUP → FILL_AR → FILL_R`
* *Read miss, dirty eviction:* `LOOKUP → EVICT_AW → EVICT_W → FILL_AR → FILL_R`
* *Write hit:* data written in-place; dirty bit set; no AXI transaction issued
* *Write miss (write-allocate):* line allocated from memory first, then modified

Writebacks use AXI INCR8 bursts (address increments each beat), since the
entire dirty line must be transferred sequentially regardless of which word
triggered the eviction.

*Cache Maintenance Operations (CMO):*

Software can issue CMO instructions to manage cache coherency for DMA
workloads.  The supported operations are:

[cols="1,3",options="header",stripes=even]
|===
| Operation    | Effect
| `INVAL`      | Invalidate a specific cache line; dirty data is *discarded*
| `FLUSH`      | Write back dirty data then invalidate the line
| `CLEAN`      | Write back dirty data but *retain* the line as valid (no eviction)
| `FLUSH_ALL`  | Flush and invalidate all lines; required before inbound DMA transfers
|===

=== Pipeline Stall Mechanics

Understanding why CPI remains above 1.0 even with a fully warm cache is
essential for interpreting the stall percentages in this report.

*Store Buffer (`SB_DEPTH = 4`):*

The KV32 pipeline decouples store instructions from the D-cache through a
4-entry store buffer (SB).  A store completes the Execute stage and is
written into a free SB slot without stalling the pipeline.  The SB drains to
the D-cache asynchronously:

* If the SB is full when a new store arrives at Execute, the pipeline stalls
  until a slot becomes free
* A `fence` instruction stalls until the SB is completely drained
* D-cache misses encountered during SB drain appear in the stall cycle count
  but are not attributable to the triggering instruction

*Instruction Buffer (`IB_DEPTH = 4`):*

The 4-entry instruction buffer decouples the Fetch stage from Decode,
allowing the I-cache to pre-fill the buffer while Decode processes earlier
instructions.  This hides the single-cycle I-cache hit latency for sequential
instruction streams and amortises I-cache refill pauses over multiple
instruction slots.

*Load-Use Hazard:*

A load instruction followed immediately by an instruction that uses the
loaded value causes a 1-cycle interlock stall.  The pipeline detects when
the Execute stage contains a load whose destination register matches a source
register of the instruction currently in Decode.

*Data Forwarding:*

To reduce load-use and data-dependence stalls, the pipeline implements two
forwarding paths:

. *MEM→ID forwarding (priority):* the result from the Memory-stage result
  register is forwarded directly to Execute inputs on the next cycle
. *WB→ID forwarding:* the value being committed at Write-Back is forwarded
  to Execute when MEM forwarding does not apply

*Why Residual CPI Exceeds 1.0:*

Even at IC-16K+DC-16K with 99%+ hit rates, CPI stays in the 1.6-2.0 range.
The residual stall sources are:

[cols="2,4",options="header",stripes=even]
|===
| Stall Source             | Root Cause
| Branch resolution        | 2-cycle penalty on each taken branch;
                             no branch predictor in the in-order pipeline
| Load-use hazard          | 1-cycle interlock per load-to-use dependence
| Store buffer pressure    | Stall when all 4 SB slots are occupied;
                             common in store-heavy loops
| `fence` / CSR access     | Serialising operations drain the pipeline and SB
| AXI handshake overhead   | Even cache-hit responses traverse the internal AXI
                             read channel; RVALID/RREADY handshake adds at least
                             one cycle per data access
| Compulsory cache misses  | Cold-start and conflict misses that no finite cache
                             size can eliminate
|===

Together these account for the 35-51% residual stall percentage visible in
the benchmark data even for the largest cache configurations tested.

=== Benchmarks

[cols="1,1,4",options="header",stripes=even]
|===
| Benchmark | Type | Behaviour and Cache Sensitivity

| `dhry`
| Control-flow / integer
| Dhrystone.  Procedure calls, string copies, conditional branches.
  Compiled code footprint ~4 KB; data working set < 1 KB.
  Instruction-dominated: I-cache has the dominant single-sided benefit.
  Working set saturates at IC-4K.

| `coremark`
| Integer / memory
| CoreMark.  List processing, matrix multiply, state machines.
  Code footprint ~8 KB; data working set 2-4 KB.
  Balanced: nearly equal benefit from I-cache and D-cache.
  Continues to improve through IC-8K.

| `embench`
| Embedded algorithms
| EMBench.  FFT, matrix multiply, bitmanip, AES kernels.
  Tight compute loops with small constant data arrays.
  Instruction-slightly-dominant: algorithm kernels fit in 4-6 KB.

| `mibench`
| Mixed workloads
| MiBench subset.  Sorting, string search, bitcount, CRC.
  Code compact (~3 KB); data arrays range 1-16 KB.
  Data-dominated: D-cache single-sided benefit outperforms I-cache.

| `rtos`
| OS kernel
| Mini-RTOS 4-test validation suite: (1) round-robin scheduler correctness,
  (2) semaphore producer/consumer, (3) mutex with priority inheritance
  (prevents priority inversion), (4) div/mul correctness under frequent
  timer preemptions.  All TCBs and per-task stacks are statically allocated
  (BSS/data); doubly-linked ready and blocked queues reference these static
  structures -- no heap is used.  Instruction-dominant at the kernel level;
  D-cache benefits from keeping scheduler queues and TCB fields warm across
  context switches.  High no-cache CPI driven by cold-start misses on the
  kernel text and TCB data at the start of each test phase.

| `cachebench`
| Bandwidth stress
| Explicit 1-16 KB dataset sweep with sequential read, write, RMW.
  Deliberately exceeds common cache sizes.
  Strongly instruction-dominated at this size (IC benefit > DC).
  The only benchmark with clear improvement from IC-16K+DC-16K over 8K+8K.
|===

All 234 simulation runs use a single compiled binary per benchmark.  No
compilation flags change between cache configurations, eliminating compiler
artefacts that could introduce cross-configuration measurement noise.

*Compilation toolchain:*

[cols="2,3",options="header",stripes=even]
|===
| Setting         | Value
| Compiler        | `riscv32-unknown-elf-gcc`
| Architecture    | `rv32imac` -- integer multiply/divide, compressed ISA (C), atomics (A)
| ABI             | `ilp32` (32-bit integers, no hardware FPU)
| Optimisation    | `-O2`
| Linker script   | `.text` at `0x10000`, `.data` at `0x20000`, stack top at `0x30000`
|===

*Benchmark-specific runtime parameters:*

[cols="2,4",options="header",stripes=even]
|===
| Benchmark    | Key Parameters and Notes

| `dhry`
| Dhrystone 2.1; 500 000 iterations; cycle count from RISC-V `mcycle` CSR;
  compiled text footprint ~3-4 KB; data working set < 1 KB.

| `coremark`
| CoreMark v1.0; `TOTAL_DATA_SIZE=2000` bytes; static working set;
  iterations auto-scaled at build time for multi-million-cycle runs.

| `embench`
| EMBench benchmark suite; kernels compiled `--without-libm`;
  individual data arrays <=2 KB; full code footprint fits in IC-4K.

| `mibench`
| Selected subset: `basicmath`, `bitcount`, `qsort`, `susan`;
  data arrays up to 8 KB; exercises D-cache at small capacity points.

| `rtos`
| Mini-RTOS 4-test suite (scheduler, semaphore, priority inheritance,
  div/mul under preemption); supervisor task (prio 2) plus up to 4
  concurrent sub-tasks spawned sequentially (`MRTOS_MAX_TASKS=8`);
  all TCBs/stacks statically allocated; tick tuned per memory back-end
  for Test 4 stress tick: SRAM dual-port = `5000 * MEM_READ_LATENCY`
  mtime ticks, DDR4 = 50 000-75 000 mtime ticks, to prevent
  ISR flood from masking cache behaviour.

| `cachebench`
| 5 dataset sizes: 1 KB, 2 KB, 4 KB, 8 KB, 16 KB (`CB_NUM_SIZES_LIMIT=5`);
  sequential read, write, and read-modify-write passes at each size;
  repetition count held constant so each dataset size contributes equally.
  The 16 KB dataset intentionally exceeds all tested cache sizes,
  guaranteeing compulsory D-cache misses at every configuration.
|===

The `cachebench` dataset sweep is uniquely sensitive to cache capacity:
small datasets fit entirely in cache and run at near-SRAM CPI; the 16 KB
dataset stresses both I-cache (sequential loop body) and D-cache (sequential
data writes), producing the widest CPI spread across configurations.

== Baseline: The Cost of Running Without Cache

Before evaluating cache configurations, it is useful to understand the
absolute penalty of running with *no cache at all*, where every instruction
fetch and every load/store access goes directly through the AXI bus to RAM.

=== CPI Without Cache

*Why SRAM stalls so much without cache:*  Even at 1-cycle AXI latency,
the KV32's burst-based cache line fill protocol issues an 8-beat burst for
each miss.  Without a cache, every instruction fetch and every load/store
triggers its own individual AXI transaction -- there is no burst amortisation.
The AXI arbiter and bus pipeline add further overhead.  The result is that
no-cache SRAM CPI is 2-2.5x rather than 1.0x.

*DDR4 latency model:*  The `ddr4_axi4_slave.sv` testbench model implements
realistic DDR4 timing (tRCD + CAS + tRP).  At these latencies, every uncached
access costs 30-50 CPU cycles, explaining the 15-19x no-cache CPI.  Note that
DDR4-1600 and DDR4-3200 produce near-identical no-cache CPI: in this
configuration, the row-access latency (measured in CPU clocks) is dominated
by fixed DDR timing parameters rather than data-rate-dependent transfer time.
The bandwidth difference between 1600 and 3200 only manifests when sustained
burst throughput is measured, not single-access latency.

=== Implication for System Design

Any serious deployment of the KV32 on DDR4 *requires* caches.  Targeting
DDR4-1600 without caches costs approximately 10x as many cycles compared
with the same memory with IC-8K+DC-8K.  The die area for an 8 KB SRAM is
recovered in milliseconds of typical runtime at these performance penalties.

== Benchmark-by-Benchmark Cache Behaviour Analysis

=== Dhrystone (`dhry`)

Dhrystone emphasises procedure calls, string copies, and integer comparisons.
Compiled text footprint is approximately 3-4 KB.

*Cache behaviour:* The I-cache captures essentially the entire program text
at IC-4K, achieving 99.88% hit rate.  The data working set is a handful of
stack variables and globals; DC-4K reaches 93.5% data hit rate.

*Key numbers:*

* No cache to IC-4K+DC-4K: CPI drops from 2.490 to 1.710 (-31%)
* 4K+4K to 8K+8K: 1.710 to 1.690 (-1.2%) -- instruction text and data
  are both nearly fully captured at 4K
* 8K+8K to 16K+16K: 1.690 to 1.680 (-0.6%) -- marginal residual miss reduction

*Recommendation:* IC-4K+DC-4K is the practical optimum.  Additional SRAM
gives diminishing returns for this workload.

=== CoreMark (`coremark`)

CoreMark exercises list processing, matrix operations, and state machines.
Code footprint is ~8 KB; data working set is 2-4 KB.

*Cache behaviour:* IC-4K achieves 99.55% I-cache hit rate; larger caches
push toward 99.88%.  D-cache achieves 99.5% at 4K for this workload because
the data structures are compact and frequently reused.  Both I-cache and
D-cache contribute across all size steps.

*Key numbers:*

* No cache to 4K+4K: 2.360 to 1.570 (-33%)
* 4K+4K to 8K+8K: 1.570 to 1.550 (-1.3%) -- I-cache captures slightly
  more of the 8 KB code footprint
* 8K+8K to 16K+16K: 1.550 to 1.540 (-0.6%)

*Recommendation:* IC-8K+DC-8K provides the sweet spot.  The 16 KB upgrade
gives less than 1% additional gain.

=== RTOS Task-Switch (`rtos`)

The `rtos` benchmark is a 4-test validation suite for the Mini-RTOS kernel
(`sw/rtos`).  A supervisor task (priority 2) creates and orchestrates the
following tests sequentially:

. *Round-robin scheduler* -- two equal-priority tasks yield to each other;
  verifies both make progress under the CLINT-driven tick.
. *Semaphore producer/consumer* -- one producer posts a semaphore twice;
  two consumer tasks must both be woken in FIFO order.
. *Mutex with priority inheritance* -- LOW (priority 4) holds a mutex while
  HIGH (priority 1) blocks on it; the kernel must raise LOW's effective
  priority to 1 so MED (priority 2) cannot starve LOW.
. *Div/mul correctness under preemption* -- two tasks run integer
  `div`/`rem`/`mul` loops while a very short tick period causes frequent
  ISR preemptions mid-operation, exercising the context-save/restore path.

All TCBs and per-task stacks are *statically allocated* (global BSS/data);
no heap is used.  The scheduler ready and blocked queues are doubly-linked
lists threading through these static TCBs.

*Cache behaviour:* IC-4K single-sided (DC-off) drops CPI from 2.350 to 2.270
(-3%).  DC-4K single-sided (IC-off) drops CPI to 2.050 (-13%), confirming
that data misses dominate.  The kernel text (~3-4 KB) fits entirely in IC-4K,
so the I-cache benefit saturates early.  D-cache gains come primarily from
keeping the scheduler queue structures, TCB fields (state, wake_tick,
eff_priority), and semaphore/mutex control blocks warm across context switches.

*Key numbers:*

* No cache to 4K+4K: 2.350 to 2.030 (-14%)
* 4K+4K to 8K+8K: 2.030 to 1.970 (-3.0%) -- D-cache improvement as the TCBs
  for all active sub-tasks fit together in a larger working set
* 8K+8K to 16K+16K: 1.970 to 1.960 (-0.5%)

*Recommendation:* IC-4K+DC-4K covers this workload well.  IC-8K+DC-8K
provides a useful margin for systems running more tasks or deeper call stacks.

=== MiBench (`mibench`)

MiBench subset includes sorting, CRC, bitcount, and string search.  Code is
compact (~3 KB); data arrays grow up to 16 KB (sort buffers, CRC tables).

*Cache behaviour:* DC single-sided (IC-off) speedup of 1.13x equals
IC single-sided 1.23x -- more balanced than classic literature suggests
at this scale.  DC-4K achieves 92.4% hit rate due to the moderate data
array sizes fitting in 4K.

*Key numbers:*

* No cache to 4K+4K: 2.250 to 1.670 (-26%)
* 4K+4K to 8K+8K: 1.670 to 1.660 (-0.6%)
* 8K+8K to 16K+16K: 1.660 to 1.660 (no change at 2-decimal precision)

*Recommendation:* IC-4K+DC-4K effectively saturates performance for this
workload.  Larger caches offer diminishing returns.

=== CacheBench (`cachebench`)

CacheBench sweeps working set sizes from 1 KB to 16 KB with sequential and
read-modify-write patterns.  It is designed to exceed typical cache sizes and
represents an upper-bound stress test rather than a realistic workload.

*Cache behaviour:* No-cache CPI is 2.210 (similar to other benchmarks on SRAM
because the DDR4 latency effect dominates only on DDR4).  IC-4K single-sided
provides speedup of 1.21x while DC-4K provides only 1.07x -- instruction-dominant
because the sweep loop itself is compact but references a data array that
exceeds the DC-4K capacity, causing many conflict and capacity misses.

*Key numbers:*

* No cache to 4K+4K: 2.210 to 1.730 (-22%)
* 4K+4K to 8K+8K: 1.730 to 1.700 (-1.7%)
* 8K+8K to 16K+16K: 1.700 to 1.670 (-1.7%) -- larger cache captures the
  full 16 KB sweep dataset

*Recommendation:* For DSP or streaming applications with large filter
buffers, IC-8K+DC-16K is the effective target.  This is the only benchmark
that continues to improve significantly at 16K+16K.

=== EMBench (`embench`)

EMBench algorithm kernels include FFT, matrix multiply, and AES.  Code per
kernel is 2-4 KB; data is local (stack-allocated or BSS) and at most 2 KB
per kernel invocation.

*Cache behaviour:* IC-4K achieves 99.27% I-cache hit rate.  D-cache achieves
88% hit rate at 4K -- lower than most benchmarks because the AES key schedule
and FFT twiddle tables exercise slightly larger data sets.

*Key numbers:*

* No cache to 4K+4K: 2.100 to 1.630 (-22%)
* 4K+4K to 8K+8K: 1.630 to 1.620 (-0.6%)
* 8K+8K to 16K+16K: 1.620 to 1.620 (no change)

*Recommendation:* IC-4K+DC-8K is a cost-effective split, improving D-cache
hit rate from 88% to ~88%, a slight improvement in conflict misses.

== Impact of Memory Back-End

The CPI values reported so far use SRAM as the memory back-end.  Real
systems often attach to DDR4, and the choice of back-end has a dramatic
multiplier effect on cache miss penalty.

=== DDR4 Timing Model

The simulation testbench models DDR4 timing using a cycle-accurate AXI slave
that enforces JEDEC standard parameters.  The following table summarises the
key timing constants for the two tested speed grades:

[cols="3,1,1,1",options="header",stripes=even]
|===
| Parameter                                 | SRAM | DDR4-1600 | DDR4-3200
| Data transfer rate (MT/s)                 | —    |      1600 |      3200
| DDR4 DRAM clock frequency (MHz)           | —    |       800 |      1600
| tCK -- DRAM clock period (ps)             | —    |      1250 |       625
| CL -- CAS latency (DDR4 full-rate cycles) | —    |        11 |        22
| tRCD -- RAS-to-CAS delay (ns)             | —    |        14 |        14
| tRP -- Row precharge time (ns)            | —    |        14 |        14
| tRCD converted to DDR4 cycles             | —    |        12 |        23
| READ_LAT = tRCD + CL (DDR4 cycles)        | —    |        23 |        45
| Approximate miss penalty (core cycles)    |  ~8  |       ~32 |       ~43
|===

*Row-open latency derivation:*

For a row-miss (the typical case for a cold cache or a conflict miss), the
memory controller must follow this sequence:

. Issue a row-activate (ACT) command; wait `tRCD` for the row sense amplifiers
  to settle
. Issue a column-read (CAS) command; wait `CL` internal pipeline cycles
. Receive the first data beat; the remaining 7 beats of the 8-beat burst
  follow at one beat per DDR4 clock

The composite row-open latency is `READ_LAT = tRCD_cycles + CL_cycles`.  For
DDR4-3200, `tRCD` in DDR4 cycles is 23 (= ceil(14 ns / 0.625 ns/cycle)) and
CL = 22 cycles, giving READ_LAT = 45 DDR4 cycles.  Although CL doubles in
cycles from DDR4-1600 to DDR4-3200, the absolute latency in nanoseconds is
unchanged (CL x tCK = 11 x 1.25 ns ~ 22 x 0.625 ns ~ 13.75 ns): DDR4-3200
runs its DRAM clock twice as fast, so the same physical latency demands twice
as many cycles.

When these DDR4 latencies are converted to core clock cycles via the
clock-domain-crossing logic in the testbench, the two DDR4 grades produce
nearly identical miss penalties -- which is precisely why the no-cache CPI
for DDR4-1600 and DDR4-3200 differs by less than 0.2 CPI in all benchmarks.

*Key takeaway:* On DDR4, the tRCD + CL + 8-beat burst dominates every miss.
This makes the miss *rate*, not the DDR4 speed grade, the primary lever for
improving DDR4-attached performance -- and is the reason caches provide a
9-10x speedup on DDR4 but only a 1.3-1.5x speedup on SRAM.

== Associativity: 2-Way vs. 4-Way

Set-associative caches use multiple ways to reduce *conflict misses* -- the
situation where two frequently accessed addresses map to the same cache set
and repeatedly evict each other.  Increasing from 2-way to 4-way doubles the
number of lines per set, allowing more address mappings to coexist without
conflict.  The trade-off is additional tag-comparison logic and slightly
longer hit latency.

This sweep added three 4-way configurations (IC-4K+DC-4K, IC-8K+DC-8K,
IC-16K+DC-16K) to quantify the practical benefit of higher associativity on
KV32 workloads.

The I-cache hit rates are already >=99.26% at 2-way.  Conflict misses in a
2-way 4K I-cache occur when two hot loops map to the same set and thrash each
other.  For these benchmarks the hot code footprints are small enough that
this rarely happens.  The `rtos` entry is the exception: the Mini-RTOS context
switch dispatcher and the task bodies hash to overlapping sets.  4-way
resolves this conflict, improving rtos I-cache hit rate from 99.26% to 99.79%
(3x fewer misses) and accounting for the observed CPI improvement.

=== Why 4-Way Rarely Helps on SRAM

The miss penalty on SRAM is ~8 CPU cycles per refill (8-beat burst at 1-cycle
AXI latency).  With already high hit rates (99%+) and a low miss penalty,
the formula:

  stall_cycles = miss_count × miss_penalty

means that even eliminating half the remaining misses via 4-way associativity
saves a negligible fraction of total stall cycles.  At DDR4 latency (30-50
cycles per miss), the same improvement would save substantially more cycles
-- and indeed the DDR4 `rtos` result shows a 5.3% improvement vs. 2.0% on SRAM.

=== Recommendation: Associativity

* *2-way is sufficient for all benchmarks except rtos.* For general-purpose
  embedded workloads on SRAM, moving from 2-way to 4-way provides < 1%
  benefit at the cost of increased tag logic area and marginally longer hit
  path.
* *4-way is worth considering for RTOS-heavy workloads on DDR4.* If the
  system runs Mini-RTOS on DDR4 and the thermal/area budget permits, 4-way at
  IC-4K+DC-4K saves 5% CPI on the task-switch microkernel workload.
* *Do not use 4-way to compensate for insufficient cache size.* If the
  working set exceeds the cache capacity, capacity misses -- not conflict
  misses -- are dominant.  Doubling the cache size (e.g., 4K to 8K) eliminates
  far more misses than quadrupling the associativity.

== Branch Prediction and RAS Sizing (IC-8K+DC-8K Baseline)

The cache sweep above recommends IC-8K+DC-8K (2-way) as the default cache
configuration.  To size the control-flow predictor structures on top of this
cache point, a dedicated predictor sweep was added in
`scripts/cache_benchmark_v2.sh` (`MODE=predictor`).

=== Methodology

* Cache fixed at IC-8K+DC-8K, 2-way, 32-byte lines
* Memory back-ends: SRAM, DDR4-1600, DDR4-3200
* Synthetic microbenchmarks:
** `btbbench`: many static branch sites to stress BTB capacity/conflicts
** `bhtbench`: mixed branch-direction patterns to stress BHT aliasing
** `rasbench`: deep nested call/return chains to stress RAS depth
* Quick sizing matrix used for this report update:
** BTB: 16, 32, 64 entries
** BHT: 32, 64, 128 entries
** RAS: 4, 8, 16 entries
** Baseline comparison against predictor-off (`BP_EN=0`, `RAS_EN=0`)
* RTL branch-predictor performance counters (added to simulation output)
  captured per run: conditional branches retired, unconditional jumps,
  taken predictions issued, misprediction redirects, and RAS push/pop counts.

Dataset source: `build/cache_bench_v2/predictor_results.csv` (99 runs + header).

=== Predictor-On Benefit

Average CPI and misprediction rate across all predictor microbenchmarks and
all three memory models (memory has negligible effect on predictor sensitivity
at this cache configuration):

[cols="3,1,1,1",options="header",stripes=even]
|===
| Configuration | Avg CPI | Mispreds / run (avg) | Accuracy
| Predictor OFF (`BP_EN=0`) | 1.5967 | — | —
| Predictor ON (`BTB=32,BHT=64,RAS=8`) | 1.5111 | 118,926 | 48.82%
|===

Net CPI reduction from predictor: *(1.5967 − 1.5111) / 1.5967 = **5.36%**.*

Per-benchmark detail (averaged across all three memory types, BTB=32/BHT=64/RAS=8):

[cols="2,1,1,1,1",options="header",stripes=even]
|===
| Benchmark | BP-off CPI | BP-on CPI | Mispreds / run | Accuracy
| `btbbench` | 1.6600 | 1.5867 | 128,722 | 37.88%
| `bhtbench` | 1.6300 | 1.5700 | 195,553 | 31.30%
| `rasbench` | 1.5000 | 1.3767 | 32,502 | 77.27%

`rasbench` shows both the largest absolute CPI gain (−8.2%) and the highest
prediction accuracy (77%), because its call/return pattern is highly regular.
`bhtbench` has the lowest accuracy (31%) due to its periodic aliasing patterns
deliberately designed to stress the 2-bit counter table.

=== RAS Depth Impact

[cols="2,1,1,1",options="header",stripes=even]
|===
| RAS depth | Avg CPI | Mispreds / run | Accuracy
| 4 | 1.5022 | 114,128 | 52.17%
| 8 | 1.5111 | 118,926 | 48.82%
| 16 | 1.4922 | 109,334 | 55.52%

The RAS effect is almost entirely concentrated in `rasbench`:

[cols="2,1,1,1",options="header",stripes=even]
|===
| RAS depth | `rasbench` CPI | `rasbench` mispreds | `rasbench` accuracy

`bhtbench` are RAS-insensitive as expected.

=== Recommendation: BTB/BHT/RAS Sizes

For the IC-8K+DC-8K default cache point:

* `BP_EN=1`
* `BTB_SIZE=64`
* `BHT_SIZE=64`
* `RAS_DEPTH=16`

Rationale:

* BTB 64 captures the full accuracy gain over BTB 32 (+2.8 pp, 51.6% vs
  48.8%) with a −0.36% CPI benefit and low area cost
  (64 × ~37 bits ≈ 296 bytes).
* BHT 64 is area-proportionate and matches larger options on every measured
  workload.
* RAS 16 cuts return mispredictions by 88.5% vs depth 8 on `rasbench`
  (97.4% accuracy vs 77.3%), yielding a 4.1% improvement in `rasbench` CPI.

If area is tightly constrained, `BTB_SIZE=32` and `RAS_DEPTH=8` remains a
reasonable fallback (48.8% accuracy overall, 5.4% CPI improvement over no
predictor) with limited loss on non-deeply-recursive workloads.

== Conclusions

. *Caches are mandatory for DDR4 systems.* Without caches, DDR4 CPI is
  15-19x.  Even IC-4K+DC-4K brings DDR4 performance within 35-50% of SRAM
  performance; IC-8K+DC-8K achieves within 5-16%.

. *IC-4K+DC-4K is the performance knee.* The first 8 KB of cache (4K+4K)
  eliminates 14-33% of the no-cache CPI on SRAM and is by far the most
  area-efficient configuration.  This is the minimum recommended configuration
  for any real deployment.

. *IC-8K+DC-8K is the recommended default.* At 16 KB total, it provides
  1-3% additional improvement over 4K+4K with well-characterised behaviour
  across all workloads.  It balances performance, area, and generality.

. *IC-16K+DC-16K offers marginal gains over 8K+8K.* The 8K to 16K doubling
  yields less than 2% CPI improvement for most benchmarks.  It is only
  justified when `cachebench`-like bandwidth workloads or large RTOS task sets
  are representative of production use.

. *Asymmetric splits provide no clear winner.* IC-8K+DC-4K and IC-4K+DC-8K
  have essentially equal average CPI for these workloads.  Both are slightly
  worse in average CPI than the symmetric IC-8K+DC-8K.  The symmetric
  configuration is recommended unless the workload is known to be strongly
  I-cache or D-cache bound.

. *4-way associativity is rarely beneficial on SRAM.* With hit rates already
  at 99%+ and an 8-cycle SRAM miss penalty, eliminating conflict misses via
  4-way saves < 2% CPI across all benchmarks.  The one exception is `rtos` at
  IC-4K+DC-4K, where 4-way resolves context-switch code conflicts for a 2%
  improvement.  On DDR4, the `rtos` improvement reaches 5.3%, making 4-way
  worth considering for RTOS-heavy DDR4 systems.

. *Stall cycles never reach zero.* Even at IC-16K+DC-16K, 35-49% stall
  fraction remains due to compulsory misses, AXI bus pipeline overhead, and
  structural hazards.  Further reductions would require hardware prefetching,
  non-blocking cache fills, or loop-body partitioning -- all beyond the scope
  of this analysis.

== Test Methodology

=== Simulation Environment

All measurements were obtained via RTL simulation of the full KV32 SoC,
including the AXI crossbar, CLINT, and memory models.

* *Simulator:* Verilator (cycle-accurate RTL simulation)
* *Memory model (SRAM):* `mem_axi.sv` with 1-cycle AXI response
* *Memory model (DDR4):* `ddr4_axi4_slave.sv` -- parameterised DDR4 timing
  model with configurable CAS latency, tRCD, and tRP
* *AXI bus width:* 32-bit data width; 8-beat bursts for cache line fills
  (32 bytes per line)

=== Metric Accounting

All metrics are computed over the full benchmark run, from first instruction
fetch to the final `ecall` that terminates the benchmark.

* *CPI:* reported directly by the KV32 simulation statistics output
* *Stall %:* `stall_cycles / total_cycles * 100`
* *I-Cache hit rate:* `(I_accesses - I_misses) / I_accesses`
* *D-Cache hit rate:* `(D_accesses - D_misses) / D_accesses`

Retired instructions are counted at WB stage, not Fetch stage.  Flushed
instructions (due to branch mispredictions or exception handling) are not
counted as retired.

=== Cache Parameters

The KV32 I-cache and D-cache share the same microarchitecture:

* *Organisation:* 2-way or 4-way set-associative (configurable)
* *Line size:* 32 bytes (8 words)
* *Replacement policy:* LRU (least-recently used)
* *D-cache write policy:* write-back with write-allocate
* *D-cache coherence:* single-core; no hardware coherence protocol

=== Benchmark Compilation

All benchmarks were compiled with RISC-V GCC (riscv32-unknown-elf-gcc)
targeting RV32IMAC with `-O2` optimisation.  The same binary was used for
all cache configurations; no benchmark-specific tuning was applied.
