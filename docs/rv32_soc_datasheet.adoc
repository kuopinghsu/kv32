== Abstract

This document provides comprehensive technical specifications for the RV32IMA System-on-Chip (SoC), a high-performance 32-bit RISC-V processor platform. The SoC features a 5-stage pipelined RV32IMA core with instruction cache, AXI4/AXI4-Lite interconnect, and a rich set of peripherals including UART, SPI, I2C, DMA, timer, and interrupt controller. Designed for embedded applications, the SoC supports both RTL simulation and FPGA implementation.

== Introduction

=== Overview

The RV32IMA SoC is a complete embedded processor system implementing the RISC-V RV32IMA instruction set architecture with Machine mode (M-mode) privilege support. The system integrates:

* **RV32IMA Processor Core**: 5-stage pipeline with instruction cache, store buffer, and instruction buffer
* **Memory Subsystem**: 2MB external RAM with AXI4 interface, configurable instruction cache
* **Interconnect**: AXI4-Lite crossbar interconnect supporting multiple master and slave devices
* **Standard Peripherals**: UART, SPI, I2C, DMA with FIFO-buffered interfaces
* **Interrupt System**: CLINT (Core-Local Interrupt) for timer/software interrupts, PLIC (Platform-Level Interrupt Controller) for external peripherals
* **Debug Support**: Magic device for simulation control and console I/O

=== System Components

==== Processor Core (rv32_core)

The RV32IMA processor core implements a 5-stage pipeline with the following key features:

* **Instruction Fetch (IF)**: Fetches instructions from I-cache or memory via instruction buffer
* **Instruction Decode (ID)**: Decodes instructions, reads register file
* **Execute (EX)**: Executes ALU operations, computes addresses, evaluates branches
* **Memory (MEM)**: Accesses data memory via store buffer
* **Writeback (WB)**: Writes results to register file, retires instructions

The core supports:

* Data forwarding to resolve RAW hazards
* Pipeline stalls for load-use dependencies
* Branch prediction (static not-taken)
* Configurable multiply/divide (fast combinatorial or slow serial)
* Instruction buffer (IB_DEPTH=4) for outstanding instruction fetches
* Store buffer (SB_DEPTH=2) for non-blocking stores

==== Instruction Cache (rv32_icache)

An optional configurable instruction cache improves performance by reducing memory latency:

* **Size**: 4KB default (parameterizable)
* **Associativity**: 2-way set-associative (parameterizable)
* **Line Size**: 32 bytes (8 words)
* **Replacement Policy**: Round-robin per set
* **Bypass Mode**: Cacheable or direct-mapped via CMO operations
* **AXI Interface**: WRAP bursts for cache-line fills (critical-word-first)

==== Memory Subsystem

* **External RAM**: 2MB AXI4 SRAM at 0x8000_0000, supports burst reads/writes
* **Instruction Arbiter**: Arbitrates between instruction fetch and data memory paths
* **AXI Crossbar**: 1-to-8 interconnect routes transactions to appropriate slaves

==== Interrupt System

* **CLINT**: Machine-mode timer (mtime/mtimecmp) and software interrupts (msip)
* **PLIC**: Routes up to 31 external interrupt sources with programmable priorities
* **Core Integration**: timer_irq, software_irq, external_irq inputs to core CSR unit

=== Address Decode Rules

The AXI crossbar (`axi_xbar`) decodes addresses based on the high-order bits:

* **Slave 0 (RAM)**: `addr[31:21] == 11'h400` (0x8000_0000)
* **Slave 1 (CLINT)**: `addr[31:20] == 12'h020` (0x0200_0000)
* **Slave 2 (PLIC)**: `addr[31:26] == 6'h03` (0x0C00_0000)
* **Slave 3 (UART)**: `addr[31:16] == 16'h2000` (0x2000_0000)
* **Slave 4 (I2C)**: `addr[31:16] == 16'h2001` (0x2001_0000)
* **Slave 5 (SPI)**: `addr[31:16] == 16'h2002` (0x2002_0000)
* **Slave 6 (DMA)**: `addr[31:12] == 20'h2003_0` (0x2003_0000)
* **Slave 7 (Magic)**: `addr[31:16] == 16'hFFFF` (0xFFFF_0000)

Unmapped addresses return an AXI SLVERR (2'b10) response on the read channel and OKAY (2'b00) on the write channel (write is silently ignored).

NOTE: The core supports Machine mode (M-mode) only. User mode (U-mode) and Supervisor mode (S-mode) are not implemented.

=== Pipeline Architecture

The core uses a 5-stage in-order pipeline:

[cols="1,2,3"]
.Pipeline Stages
|===
|Stage |Name |Function

|IF
|Instruction Fetch
|Request instruction from I-cache/memory via instruction buffer

|ID
|Instruction Decode
|Decode opcode, read source registers, generate control signals

|EX
|Execute
|Execute ALU operation, compute memory address, evaluate branch condition

|MEM
|Memory Access
|Perform load/store via data memory interface and store buffer

|WB
|Writeback
|Write result to destination register, retire instruction
|===

==== Instruction Buffer (IB)

The instruction buffer decouples instruction fetch from pipeline execution:

* **Depth**: IB_DEPTH (default 4, parameterizable)
* **Purpose**: Allow multiple outstanding instruction fetches to hide memory latency
* **Benefit**: Keeps pipeline fed even during I-cache misses

==== Store Buffer (SB)

The store buffer allows stores to complete asynchronously:

* **Depth**: SB_DEPTH (default 2, parameterizable)
* **Purpose**: Buffer store requests so pipeline doesn't stall on store completion
* **Ordering**: Maintains program order for stores

==== Forwarding and Hazards

* **Data Forwarding**: EX→EX, MEM→EX, WB→EX paths resolve RAW hazards
* **Load-Use Stall**: Pipeline stalls 1 cycle when load result is immediately used
* **Branch Prediction**: Static not-taken; 2-cycle penalty on taken branches (flush IF and ID stages)
* **Structural Hazards**: Single memory port shared between I-fetch and D-access via arbiter

==== Trap Handling

On exception or interrupt:

1. PC is saved to `mepc`
2. Trap cause is written to `mcause`
3. Trap value (e.g., faulting address) is written to `mtval`
4. `mstatus.MIE` is cleared (disable interrupts)
5. PC is set to `mtvec` (trap vector address)

Return from trap: `MRET` instruction restores `mepc` to PC and restores interrupt enable.

== Instruction Cache Specification

=== Overview

The instruction cache (rv32_icache) reduces instruction fetch latency by caching frequently accessed code:

[cols="1,3"]
.I-Cache Configuration
|===
|Parameter |Default Value

|CACHE_SIZE
|4096 bytes (4KB)

|CACHE_LINE_SIZE
|32 bytes (8 words)

|CACHE_WAYS
|2 (2-way set-associative)

|Number of Sets
|64 (computed: SIZE / (LINE_SIZE × WAYS))

|Replacement Policy
|Round-robin (per-set victim pointer)

|Write Policy
|N/A (instruction cache is read-only)
|===

=== Cache Organization

* **Address Decomposition**:
** Tag: bits [31:11] (21 bits)
** Index: bits [10:5] (6 bits, selects 1 of 64 sets)
** Word Offset: bits [4:2] (3 bits, selects 1 of 8 words)
** Byte Offset: bits [1:0] (always 00 for aligned word access)

* **Storage**:
** Valid bits: flip-flops (2 ways × 64 sets = 128 bits)
** Tag storage: SRAM macros (2 instances, 64 entries × 21 bits)
** Data storage: SRAM macros (2 instances, 512 entries × 32 bits)

=== Cache Operations

==== Normal Fetch (Cache Hit)

1. Core asserts `imem_req_valid` with address
2. Cache performs lookup (1 cycle)
3. On hit, returns data via `imem_resp_data` next cycle
4. Core accepts response and continues

**Latency**: 1 cycle (registered output)

==== Cache Miss

1. Cache detects miss in lookup stage
2. Issues AXI WRAP burst for entire cache line (8 beats)
3. Returns critical word first (early restart) to unblock core
4. Remaining beats drain in background while core continues
5. Victim way is selected via round-robin pointer

**Latency**: Memory latency + 1 cycle (critical-word-first optimization)

==== Bypass Mode

When cache is disabled via CMO (Cache Management Operation):

* Every fetch issues single-beat AXI INCR transaction
* No caching occurs, data is forwarded directly to core
* Useful for debugging or non-cacheable code regions

=== Cache Management Operations (CMO)

The cache supports software-initiated management via the `cmo_*` sideband interface:

[cols="1,2,4"]
.Supported CMO Operations
|===
|Opcode |Name |Description

|2'b00
|CMO_INVAL
|Invalidate specific cache line matching address

|2'b01
|CMO_DISABLE
|Disable cache, enter bypass mode

|2'b10
|CMO_ENABLE
|Re-enable cache

|2'b11
|CMO_FLUSH_ALL
|Invalidate all cache entries (FENCE.I support)
|===

NOTE: CMO operations are typically triggered by `FENCE.I` or `cbo.inval` instructions in the core's MEM stage.

=== Performance Counters (Simulation Only)

The cache provides performance monitoring signals (not synthesized):

* `perf_req_cnt`: Total fetch requests
* `perf_hit_cnt`: Cache hits
* `perf_miss_cnt`: Cache misses
* `perf_bypass_cnt`: Bypass fetches (cache disabled)
* `perf_fill_cnt`: Completed line fills
* `perf_cmo_cnt`: CMO operations executed

==== Baud Rate Configuration

The UART baud rate is determined by the SoC parameter `BAUD_RATE`:

----
CLKS_PER_BIT = CLK_FREQ / BAUD_RATE
----

**Hardware Constraint**: `CLKS_PER_BIT >= 4` (absolute minimum for RX sampling).

**Recommended**: `CLKS_PER_BIT >= 8` for production use (provides timing margin).

==== Interrupt Behavior

* **RX Interrupt**: Fires when RX FIFO transitions from empty to non-empty
* **TX Interrupt**: Fires when TX FIFO becomes empty (all data transmitted)

Software should:

* **RX ISR**: Read all available bytes from DATA register until STATUS[2]=0
* **TX ISR**: Refill TX FIFO by writing to DATA register

===== Write Transaction

1. Write slave address (shifted left, bit 0=0 for write) to TX register

==== Transfer Modes

===== 1D Transfer (Linear)

Simple contiguous memory copy:

* Set `SRC_ADDR`, `DST_ADDR`, `XFER_CNT`
* Set `CTRL[4:3]=00` (MODE=1D)
* Set `CTRL[5:6]` for address increment behavior
* Set `CTRL[1]=1` to start

===== 2D Transfer (Strided)

Copy 2D arrays with row stride:

* Set `SRC_ADDR`, `DST_ADDR`, `XFER_CNT` (bytes per row)
* Set `SRC_STRIDE`, `DST_STRIDE` (bytes between row starts)
* Set `ROW_CNT` (number of rows)
* Set `CTRL[4:3]=01` (MODE=2D)

Example: Copy 16×16 pixel subregion from 640×480 framebuffer:

----
XFER_CNT = 16 * 4        // 16 pixels × 4 bytes/pixel
SRC_STRIDE = 640 * 4     // Source pitch
DST_STRIDE = 16 * 4      // Destination pitch (packed)
ROW_CNT = 16             // 16 rows
----

===== 3D Transfer (Planar)

Copy 3D volumes with plane stride:

* Configure as 2D, plus set `SRC_PLANE_STRIDE`, `DST_PLANE_STRIDE`, `PLANE_CNT`
* Set `CTRL[4:3]=10` (MODE=3D)

===== Scatter-Gather Transfer

Chain multiple transfers from descriptor list:

* Each descriptor is 16 bytes: `{src_addr, dst_addr, xfer_cnt, mode_ctrl}`
* Set `SG_ADDR` to descriptor list base address
* Set `SG_CNT` to number of descriptors
* Set `CTRL[4:3]=11` (MODE=SG)

==== Interrupt Behavior

* Each channel can raise interrupt on completion (DONE) or error (ERR)
* Channel interrupts are aggregated to single PLIC source (ID 4)
* Software reads `IRQ_STAT` to determine which channel(s) triggered
* Write 1 to `IRQ_STAT` bits to acknowledge

==== Performance Monitoring

Enable performance counters to measure DMA throughput:

----
DMA_PERF_CTRL = 0x3;    // Enable counters and reset
// ... run workload ...
cycles = DMA_PERF_CYCLES;
rd_bytes = DMA_PERF_RD_BYTES;
wr_bytes = DMA_PERF_WR_BYTES;
bandwidth_MBps = (rd_bytes + wr_bytes) / cycles * CLK_FREQ / 1e6;
----

==== AXI Interface

The DMA controller has a dedicated AXI4 master port (32-bit data width default):

* Burst optimization: DMA controller generates INCR bursts up to `MAX_BURST_LEN` beats
* Page boundary handling: Bursts never cross 4KB AXI page boundaries
* Error handling: AXI SLVERR/DECERR sets channel STAT[2]=1 and aborts transfer

==== Timer Operation

The 64-bit `mtime` counter increments by 1 every clock cycle. Timer interrupt fires when:

----
mtime >= mtimecmp
----

Software clears timer interrupt by writing a new (larger) value to `mtimecmp`.

Example: Set timer for 1ms from now (100 MHz clock):

[source,c]
----
uint64_t mtime = read_mtime();

write_mtimecmp(mtimecmp);
----

==== Software Interrupt Operation

Write `MSIP[0] = 1` to trigger machine software interrupt immediately. Clear by writing `MSIP[0] = 0`.

Typical use: Inter-processor interrupt (IPI) on multi-hart systems, or self-interrupt for context switching.

==== Interrupt Flow

1. **Pending**: Peripheral asserts `irq_src[i]`, PLIC sets `PENDING[i]=1`
2. **Arbitration**: PLIC selects highest-priority enabled pending interrupt where `priority[i] > threshold`
3. **Notification**: PLIC asserts `external_irq` to core
4. **Claim**: ISR reads `CLAIM` register to get source ID, `PENDING[i]` remains set
5. **Service**: ISR handles the interrupt, clears condition in peripheral
6. **Complete**: ISR writes source ID to `COMPLETE` register
7. **Clear**: If peripheral de-asserted `irq_src[i]`, PLIC clears `PENDING[i]`

NOTE: If peripheral IRQ remains asserted after complete, `PENDING[i]` is immediately re-set (level-sensitive behavior).

==== Priority Arbitration

When multiple interrupts are pending:

* Highest priority wins (priority 7 > priority 1)
* On priority tie, lowest source ID wins (source 1 > source 2)

Example configuration:

----
PRIORITY[1] = 5;  // UART high priority
PRIORITY[2] = 3;  // SPI medium priority
PRIORITY[3] = 3;  // I2C medium priority
PRIORITY[4] = 1;  // DMA low priority
THRESHOLD = 0;    // Allow all enabled interrupts
ENABLE = 0x1E;    // Enable sources 1-4
----

=== Crossbar Architecture

The AXI crossbar (`axi_xbar`) routes transactions from the core to appropriate slave devices:

* **Master Port**: Connected to memory arbiter (combines I-fetch + D-access paths)
* **Slave Ports**: 8 devices (RAM, CLINT, PLIC, UART, I2C, SPI, DMA, Magic)

==== Address Decode Logic

Decoding is priority-based (higher-numbered slaves checked first):

[source,systemverilog]
----
if (addr[31:16] == 16'hFFFF)       → Slave 7 (Magic)
else if (addr[31:12] == 20'h2003_0) → Slave 6 (DMA)
else if (addr[31:16] == 16'h2002)  → Slave 5 (SPI)
else if (addr[31:16] == 16'h2001)  → Slave 4 (I2C)
else if (addr[31:16] == 16'h2000)  → Slave 3 (UART)
else if (addr[31:26] == 6'h03)     → Slave 2 (PLIC)
else if (addr[31:20] == 12'h020)   → Slave 1 (CLINT)
else if (addr[31:21] == 11'h400)   → Slave 0 (RAM)
else                               → SLVERR (unmapped)
----

==== Error Responses

* **Unmapped address read**: Crossbar returns `SLVERR` response with data = 0xDEADBEEF
* **Unmapped address write**: Crossbar returns `OKAY` response (write silently dropped)

=== AXI4 Memory Interface

The external RAM interface uses full AXI4 with burst support:

[cols="1,3"]
.AXI4 Memory Features
|===
|Feature |Specification

|Data Width
|32 bits

|Burst Length
|1-256 beats (ARLEN/AWLEN encoding)

|Burst Types
|INCR (normal access), WRAP (I-cache line fills)

|Burst Size
|4 bytes per beat (ARSIZE=3'b010)

|Address Alignment
|4-byte aligned

|Write Strobes
|4 bits (byte-level granularity)
|===

==== I-Cache Burst Transactions

The instruction cache uses WRAP bursts for critical-word-first cache-line fills:

* Burst type: `ARBURST = 2'b10` (WRAP)
* Burst length: `ARLEN = (line_size / 4) - 1` (e.g., 7 for 32-byte line)
* Starting address: Critical word address (PC)
* Wrap boundary: Cache-line boundary

Example: Fetch from address 0x80000014 with 32-byte cache line:

----
ARADDR = 0x80000014
ARLEN = 7           // 8 beats
ARBURST = WRAP
Beats returned:
  0: 0x80000014 (critical word, returned first)
  1: 0x80000018
  2: 0x8000001C
  3: 0x80000000 (wrap to line boundary)
  4: 0x80000004
  5: 0x80000008
  6: 0x8000000C
  7: 0x80000010
----

==== Data Access Transactions

Data loads/stores use single-beat INCR bursts:

* Burst type: `ARBURST = 2'b01` (INCR)
* Burst length: `ARLEN = 0` (single beat)
* Write strobes: Indicate which bytes are valid (1, 2, or 4 bytes)

== Programming Guide

=== Boot Sequence

The processor boots from address `0x8000_0000` (RAM base) on reset:

1. Reset vector: PC = 0x8000_0000
2. Bootloader/firmware executes (typically copied to RAM by JTAG or external loader)
3. Initialize stack pointer
4. Initialize CSRs (mstatus, mtvec, mie)
5. Initialize peripherals (UART, timers, etc.)
6. Enable interrupts (clear mstatus.MIE)
7. Jump to main application

=== Memory Initialization

For simulation, the ELF loader (`elfloader.cpp`) loads program sections into RAM via DPI-C before simulation starts. For FPGA, use:

* **JTAG**: Load program via JTAG debugger (OpenOCD, etc.)
* **Boot ROM**: Store bootloader in on-chip ROM, copy program from external flash to RAM

=== Interrupt Handling

==== Setting Up Interrupts

[source,c]
----
#include "csr.h"
#include "rv_platform.h"
#include "rv_plic.h"
#include "rv_clint.h"

void setup_interrupts(void) {
    // Set trap vector (must be 4-byte aligned)
    write_csr(mtvec, (uint32_t)trap_entry);

    // Enable M-mode interrupts in mstatus
    set_csr(mstatus, MSTATUS_MIE);

    // Enable specific interrupt types in mie
    set_csr(mie, MIE_MEIE | MIE_MTIE | MIE_MSIE);

    // Configure PLIC
    rv_plic_set_priority(RV_PLIC_SRC_UART, 5);
    rv_plic_enable_interrupt(RV_PLIC_SRC_UART);
    rv_plic_set_threshold(0);

    // Configure CLINT timer for periodic interrupt (1ms)
    uint64_t mtime = rv_clint_read_mtime();
    rv_clint_write_mtimecmp(mtime + 100000);  // +1ms @ 100MHz
}
----

==== Trap Handler Example

[source,c]
----
void trap_handler(void) {
    uint32_t mcause = read_csr(mcause);
    uint32_t mepc = read_csr(mepc);

    if (mcause & 0x80000000) {
        // Interrupt
        uint32_t irq_code = mcause & 0x1F;
        switch (irq_code) {
        case 7:  // Timer interrupt
            handle_timer_interrupt();
            break;
        case 11: // External interrupt (PLIC)
            handle_external_interrupt();
            break;
        case 3:  // Software interrupt
            handle_software_interrupt();
            break;
        }
    } else {
        // Exception
        handle_exception(mcause, mepc);
    }
}

void handle_timer_interrupt(void) {
    // Advance mtimecmp for next interrupt
    uint64_t mtimecmp = rv_clint_read_mtimecmp();
    rv_clint_write_mtimecmp(mtimecmp + 100000);  // +1ms
}

void handle_external_interrupt(void) {
    // Claim interrupt from PLIC
    uint32_t source = rv_plic_claim_interrupt();

    switch (source) {
    case RV_PLIC_SRC_UART:
        uart_isr();
        break;
    case RV_PLIC_SRC_SPI:
        spi_isr();
        break;
    // ... handle other sources ...
    }

    // Complete interrupt
    rv_plic_complete_interrupt(source);
}
----

==== SPI Flash Read

[source,c]
----
#include "rv_spi.h"

#define SPI_CMD_READ 0x03

void spi_read_flash(uint32_t addr, uint8_t *buf, size_t len) {
    rv_spi_init(1000000, RV_SPI_MODE0);  // 1 MHz, Mode 0

    rv_spi_select(0);  // Assert CS[0]

    // Send READ command + 24-bit address
    rv_spi_transfer(SPI_CMD_READ);
    rv_spi_transfer((addr >> 16) & 0xFF);
    rv_spi_transfer((addr >> 8) & 0xFF);
    rv_spi_transfer(addr & 0xFF);

    // Read data
    for (size_t i = 0; i < len; i++) {
        buf[i] = rv_spi_transfer(0x00);
    }

    rv_spi_deselect(0);  // De-assert CS[0]
}
----

==== I2C EEPROM Write

[source,c]
----
#include "rv_i2c.h"

#define EEPROM_ADDR 0x50  // 7-bit I2C address

void i2c_eeprom_write(uint8_t mem_addr, uint8_t data) {
    rv_i2c_init(100000);  // 100 kHz

    rv_i2c_start();
    rv_i2c_write(EEPROM_ADDR << 1);  // Address + W bit
    rv_i2c_write(mem_addr);           // Memory address
    rv_i2c_write(data);               // Data byte
    rv_i2c_stop();

    // Wait for write cycle to complete (typical 5ms)
    for (volatile int i = 0; i < 500000; i++);
}
----

void dma_memcpy(void *dst, const void *src, size_t len) {

    }

=== Performance Optimization Tips

1. **Enable I-Cache**: Set `ICACHE_EN=1` parameter for significant speedup (~2-5× depending on workload)
2. **Increase Buffer Depths**: Larger `IB_DEPTH` and `SB_DEPTH` improve throughput in memory-intensive code
3. **Use DMA for Bulk Transfers**: Offload large memory copies to DMA controller
4. **Minimize Load-Use Dependencies**: Reorder instructions to avoid pipeline stalls
5. **Use Fast Multiply/Divide**: Set `FAST_MUL=1` and `FAST_DIV=1` for single-cycle operations
6. **Align Data Structures**: 4-byte alignment avoids misaligned access penalties
7. **Batch Peripheral I/O**: Use FIFOs to buffer multiple bytes before checking status
