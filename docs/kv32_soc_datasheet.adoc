== Abstract

This document provides comprehensive technical specifications for the RV32IMAC System-on-Chip (SoC), a high-performance 32-bit RISC-V processor platform. The SoC features a 5-stage pipelined RV32IMAC core with separate instruction and data caches, AXI4/AXI4-Lite interconnect, and a rich set of peripherals including UART, SPI, I2C, DMA, GPIO, timer/PWM, and interrupt controller. Runtime-configurable Physical Memory Attributes (PMA) control per-region cacheability for both caches. Designed for embedded applications, the SoC supports both RTL simulation and FPGA implementation.

== Introduction

=== Overview

The RV32IMAC SoC is a complete embedded processor system implementing the RISC-V RV32IMAC instruction set architecture with Machine mode (M-mode) privilege support. The system integrates:

* **RV32IMAC Processor Core**: 5-stage pipeline with RVC expander, instruction cache, data cache, store buffer, and instruction buffer
* **Memory Subsystem**: 2MB external RAM with AXI4 interface, configurable instruction and data caches
* **Physical Memory Attributes (PMA)**: 8 runtime-configurable regions controlling I-cache (X bit) and D-cache (C bit) cacheability per address range
* **Interconnect**: AXI4-Lite crossbar interconnect supporting multiple master and slave devices
* **Standard Peripherals**: UART, SPI, I2C, DMA with FIFO-buffered interfaces
* **Interrupt System**: CLINT (Core-Local Interrupt) for timer/software interrupts, PLIC (Platform-Level Interrupt Controller) for external peripherals
* **Debug Support**: Magic device for simulation control and console I/O

=== System Components

==== Processor Core (kv32_core)

The RV32IMAC processor core implements a 5-stage pipeline with the following key features:

* **Instruction Fetch (IF)**: Fetches instructions from I-cache or memory via instruction buffer
* **Instruction Decode (ID)**: Decodes instructions, reads register file
* **Execute (EX)**: Executes ALU operations, computes addresses, evaluates branches
* **Memory (MEM)**: Accesses data memory via store buffer
* **Writeback (WB)**: Writes results to register file, retires instructions

The core supports:

* Data forwarding to resolve RAW hazards
* Pipeline stalls for load-use dependencies
* Branch prediction (static not-taken)
* Configurable multiply/divide (fast combinatorial or slow serial)
* Instruction buffer (IB_DEPTH=4) for outstanding instruction fetches
* Store buffer (SB_DEPTH=2) for non-blocking stores

==== Instruction Cache (kv32_icache)

An optional configurable instruction cache improves performance by reducing memory latency:

* **Size**: 4KB default (parameterizable)
* **Associativity**: 2-way set-associative (parameterizable)
* **Line Size**: 64 bytes
* **Replacement Policy**: Round-robin per set
* **Bypass Mode**: Cacheable or direct-mapped via CMO operations; PMA X-bit per region
* **AXI Interface**: WRAP bursts for cache-line fills (critical-word-first)

==== Data Cache (kv32_dcache)

An optional configurable data cache reduces load/store latency:

* **Size**: 4KB default (parameterizable)
* **Associativity**: 2-way set-associative (parameterizable)
* **Line Size**: 32 bytes (8 words)
* **Write Policy**: Write-back (default) or write-through
* **Allocate Policy**: Write-allocate on store miss (default) or no-allocate (bypass)
* **Replacement Policy**: Round-robin per set
* **Dirty Eviction**: Automatically write-back dirty line before fill (write-back mode)
* **AXI Interface**: WRAP bursts for cache-line fills; AW/W/B channels for eviction and write-through
* **CMO Support**: INVAL, FLUSH, CLEAN, FLUSH_ALL operations
* **PMA Integration**: Per-region C bit controls D-cacheability; non-cacheable accesses bypass the cache

==== Physical Memory Attributes (PMA)

The PMA subsystem provides per-region cacheability control at runtime:

* **Regions**: 8 independently configurable regions
* **Address Modes**: OFF (disabled), TOR (top-of-range), NA4 (4B natural align), NAPOT (power-of-two)
* **Per-region Attributes**: X bit (I-cacheable), C bit (D-cacheable), B bit (bufferable), L bit (locked)
* **CSRs**: `pmacfg0` (0x7C0), `pmacfg1` (0x7C1), `pmaaddr0`–`pmaaddr7` (0x7C4–0x7CB)
* **Fallback Rule**: When no region matches, `addr[31]=1` → cacheable (RAM); `addr[31]=0` → non-cacheable (I/O)

==== Memory Subsystem

* **External RAM**: 2MB AXI4 SRAM at 0x8000_0000, supports burst reads/writes
* **Instruction Arbiter**: Arbitrates between instruction fetch and data memory paths
* **AXI Crossbar**: 1-to-11 interconnect routes transactions to appropriate slaves

==== Peripheral Subsystem

Seven peripheral slaves hang off the AXI crossbar:

* **UART** (`axi_uart.sv`): 8N1 serial, 16-deep TX/RX FIFOs, programmable baud rate
* **I2C** (`axi_i2c.sv`): Master-mode, 100k/400k, 8-deep TX/RX FIFOs
* **SPI** (`axi_spi.sv`): Master-mode, SPI modes 0-3, 4 CS lines, 8-deep TX/RX FIFOs
* **DMA** (`axi_dma.sv`): 4 channels, 1D/2D/3D/SG transfers, performance counters
* **GPIO** (`axi_gpio.sv`): 4 banks × 32 pins, direction/IRQ/edge/level config, loopback
* **Timer** (`axi_timer.sv`): 4 channels, 32-bit up-counter, dual compare, PWM output
* **Magic**: Simulation-only console I/O and exit-code control

==== Interrupt System

* **CLINT**: Machine-mode timer (mtime/mtimecmp) and software interrupts (msip)
* **PLIC**: Routes up to 31 external interrupt sources (UART, SPI, I2C, DMA, GPIO, Timer, WDT) with programmable priorities (0-15) and a configurable claim/complete handshake
* **Core Integration**: timer_irq, software_irq, external_irq inputs to core CSR unit

==== Power Manager (kv32_pm)

The `kv32_pm` module implements clock gating for the processor core, enabling low-power idle when the core executes a `WFI` (Wait-For-Interrupt) instruction.

===== Operation

When the processor executes `WFI`, `kv32_core` asserts `core_sleep_o` after all outstanding instruction-fetch and store-buffer traffic drains. `kv32_pm` then gates the core clock until any M-mode interrupt becomes active:

[source,verilog]
----
clk_en = !sleep_req_i       // core not in WFI
       | timer_irq_i         // CLINT machine-timer
       | external_irq_i      // PLIC external IRQ
       | software_irq_i      // CLINT software IRQ
       | irq_wake_pending;   // captured single-cycle pulse
----

===== IRQ Pulse Capture

Interrupt sources may assert for only a single cycle on the ungated system clock. To prevent the core from remaining asleep after a missed short pulse, a flip-flop clocked by the ungated `clk_i` captures any IRQ edge and holds `clk_en=1` (`irq_wake_pending`) until the core de-asserts `sleep_req_i`. This guarantees the gated clock delivers at least one rising edge to the core's interrupt-sampling logic.

===== Technology Mapping

[cols="1,3"]
.Clock-Gate Implementation
|===
|Target |Implementation

|FPGA (Xilinx UltraScale+)
|`BUFGCE` primitive with `CE_TYPE="SYNC"`. Vivado infers correct setup/hold constraints on CE. CE active-high; not inverted.

|Simulation / ASIC
|Behavioural ICG cell: transparent latch (open on low phase of `clk_i`) followed by AND gate. Recognised by synthesis tools as a standard ICG pattern.
|===

===== Timing

* **Sleep entry**: `core_sleep_o` asserted after all outstanding AXI transactions drain
* **Wake latency**: 1 clock period from IRQ assertion to first gated rising edge
* **Clock domain boundary**: CLINT and PLIC run on the ungated system clock and continue counting while the core is asleep

=== Address Decode Rules

The AXI crossbar (`axi_xbar`) decodes addresses based on the high-order bits:

* **Slave 0 (RAM)**: `addr[31:21] == 11'h400` (0x8000_0000)
* **Slave 1 (Magic)**: `addr[31:16] == 16'h4000` (0x4000_0000)
* **Slave 2 (CLINT)**: `addr[31:20] == 12'h020` (0x0200_0000)
* **Slave 3 (PLIC)**: `addr[31:24] == 8'h0C` (0x0C00_0000)
* **Slave 4 (DMA)**: `addr[31:16] == 16'h2000` (0x2000_0000)
* **Slave 5 (UART)**: `addr[31:16] == 16'h2001` (0x2001_0000)
* **Slave 6 (I2C)**: `addr[31:16] == 16'h2002` (0x2002_0000)
* **Slave 7 (SPI)**: `addr[31:16] == 16'h2003` (0x2003_0000)
* **Slave 8 (Timer)**: `addr[31:16] == 16'h2004` (0x2004_0000)
* **Slave 9 (GPIO)**: `addr[31:16] == 16'h2005` (0x2005_0000)
* **Slave 10 (WDT)**: `addr[31:16] == 16'h2006` (0x2006_0000)

Unmapped addresses return an AXI SLVERR (2'b10) response on the read channel and OKAY (2'b00) on the write channel (write is silently ignored).

== Processor Core Specification

=== RV32IMAC ISA Support

The processor core implements the following RISC-V instruction set extensions:

[cols="1,4"]

|===
|Extension |Description

|RV32I
|Base integer instruction set (32-bit)

|M
|Integer multiplication and division

|A
|Atomic instructions (LR/SC, AMO)

|C (Zca)
|16-bit compressed instructions; expanded to 32-bit by `kv32_rvc` before the decode stage

|Zicsr
|CSR access instructions (CSRRW, CSRRS, CSRRC, etc.)

|Instruction-fetch fence (FENCE.I)

NOTE: The core supports Machine mode (M-mode) only. User mode (U-mode) and Supervisor mode (S-mode) are not implemented.

=== Pipeline Architecture

The core uses a 5-stage in-order pipeline:

[cols="1,2,3"]
.Pipeline Stages
|===
|Stage |Name |Function

|IF
|Instruction Fetch
|Request instruction from I-cache/memory via instruction buffer

|ID
|Instruction Decode
|Decode opcode, read source registers, generate control signals

|EX
|Execute
|Execute ALU operation, compute memory address, evaluate branch condition

|MEM
|Memory Access
|Perform load/store via data memory interface and store buffer

|WB
|Writeback
|Write result to destination register, retire instruction
|===

==== Instruction Buffer (IB)

The instruction buffer decouples instruction fetch from pipeline execution:

* **Depth**: IB_DEPTH (default 4, parameterizable)
* **Purpose**: Allow multiple outstanding instruction fetches to hide memory latency
* **Benefit**: Keeps pipeline fed even during I-cache misses

==== Store Buffer (SB)

The store buffer allows stores to complete asynchronously:

* **Depth**: SB_DEPTH (default 2, parameterizable)
* **Purpose**: Buffer store requests so pipeline doesn't stall on store completion
* **Ordering**: Maintains program order for stores

==== Forwarding and Hazards

* **Data Forwarding**: EX→EX, MEM→EX, WB→EX paths resolve RAW hazards
* **Load-Use Stall**: Pipeline stalls 1 cycle when load result is immediately used
* **Branch Prediction**: Static not-taken; 2-cycle penalty on taken branches (flush IF and ID stages)
* **Structural Hazards**: Single memory port shared between I-fetch and D-access via arbiter

==== Trap Handling

On exception or interrupt:

1. PC is saved to `mepc`
2. Trap cause is written to `mcause`
3. Trap value (e.g., faulting address) is written to `mtval`
4. `mstatus.MIE` is cleared (disable interrupts)
5. PC is set to `mtvec` (trap vector address)

Return from trap: `MRET` instruction restores `mepc` to PC and restores interrupt enable.

=== Cache Organization

* **Address Decomposition**:
** Tag: bits [31:11] (21 bits)
** Index: bits [10:5] (6 bits, selects 1 of 64 sets)
** Word Offset: bits [4:2] (3 bits, selects 1 of 8 words)
** Byte Offset: bits [1:0] (always 00 for aligned word access)

* **Storage**:
** Valid bits: flip-flops (2 ways × 64 sets = 128 bits)
** Tag storage: SRAM macros (2 instances, 64 entries × 21 bits)
** Data storage: SRAM macros (2 instances, 512 entries × 32 bits)

=== Cache Operations

==== Normal Fetch (Cache Hit)

1. Core asserts `imem_req_valid` with address
2. Cache performs lookup (1 cycle)
3. On hit, returns data via `imem_resp_data` next cycle
4. Core accepts response and continues

**Latency**: 1 cycle (registered output)

==== Cache Miss

1. Cache detects miss in lookup stage
2. Issues AXI WRAP burst for entire cache line (8 beats)
3. Returns critical word first (early restart) to unblock core
4. Remaining beats drain in background while core continues
5. Victim way is selected via round-robin pointer

**Latency**: Memory latency + 1 cycle (critical-word-first optimization)

==== Bypass Mode

When cache is disabled via CMO (Cache Management Operation):

* Every fetch issues single-beat AXI INCR transaction
* No caching occurs, data is forwarded directly to core
* Useful for debugging or non-cacheable code regions

== Data Cache Specification

=== Overview

The data cache (`kv32_dcache`) reduces load/store latency by caching frequently accessed data. It supports both write-back and write-through policies with write-allocate or no-allocate behaviour on store misses.

[cols="1,3"]
.D-Cache Configuration
|===
|Parameter |Default Value

|DCACHE_SIZE
|4096 bytes (4KB)

|DCACHE_LINE_SIZE
|32 bytes (8 words)

|DCACHE_WAYS
|2 (2-way set-associative)

|Number of Sets
|64 (computed: SIZE / (LINE_SIZE × WAYS))

|Replacement Policy
|Round-robin (per-set victim pointer)

|Write Policy
|Write-back (DCACHE_WRITE_BACK=1); write-through with DCACHE_WRITE_BACK=0

|Allocate Policy
|Write-allocate (DCACHE_WRITE_ALLOC=1); no-allocate with DCACHE_WRITE_ALLOC=0
|===

=== Cache Organisation

* **Address Decomposition**:
** Tag: bits [31:11] (21 bits)
** Index: bits [10:5] (6 bits, selects 1 of 64 sets)
** Word Offset: bits [4:2] (3 bits, selects 1 of 8 words)
** Byte Offset: bits [1:0] (byte within word)

* **Storage**:
** Valid bits: flip-flops (2 ways × 64 sets = 128 bits)
** Dirty bits: flip-flops (2 ways × 64 sets = 128 bits, write-back mode only)
** Tag storage: SRAM macros (2 instances, 64 entries × 21 bits)
** Data storage: SRAM macros (2 instances, 512 entries × 32 bits, byte-enable write)

=== Cache Operations

==== Load Hit

1. Core asserts `core_req_valid` with address and `mem_read=1`
2. Cache performs lookup (1 cycle, SRAM read launched speculatively in `S_IDLE`)
3. On hit, returns data via `core_resp_data` next cycle (zero-stall if core is ready)

**Latency**: 1 cycle (registered output)

==== Load Miss — Clean Victim

1. Cache detects miss in lookup stage
2. Issues AXI WRAP burst for entire cache line (8 beats, critical-word-first)
3. Returns critical word early to unblock core
4. Remaining beats drain in background

**Latency**: Memory latency + 1 cycle (critical-word-first optimisation)

==== Load Miss — Dirty Victim (Write-Back Mode)

1. Cache detects miss; victim line is dirty
2. Writes dirty line back to memory (AXI write: AW → W × 8 → B)
3. Fetches new line with AXI WRAP burst
4. Returns critical word to core

**Latency**: Memory latency × 2 (eviction + fill round-trip)

==== Store Hit

1. Cache detects hit; data SRAM performs read-modify-write in `S_HIT_WR`
2. For write-back: dirty bit set; no AXI traffic
3. For write-through: dirty bit unchanged; AXI write issued (`S_WT_AW → S_WT_W → S_WT_B`)

**Write-Back Latency**: 1 cycle (no AXI traffic)
**Write-Through Latency**: Memory latency

==== Store Miss

* **Write-Allocate** (`DCACHE_WRITE_ALLOC=1`): Allocate new line (evict dirty if needed), fill from memory, then perform hit-store.
* **No-Allocate** (`DCACHE_WRITE_ALLOC=0`): Write directly to memory via AXI — no line allocation.

==== PMA Bypass (Non-Cacheable Access)

When `use_dcache=0` (PMA C bit clear or CMO_DISABLE):

* Reads: single-beat AXI INCR read, data forwarded directly to core
* Writes: single-beat AXI INCR write with byte strobes

=== Cache Management Operations (CMO)

The cache supports software-initiated management via the `cmo_*` sideband interface:

[cols="1,2,4"]
.Supported CMO Operations
|===
|Opcode |Name |Description

|2'b00
|CMO_INVAL
|Invalidate specific cache line matching address (clears valid bit; no writeback)

|2'b01
|CMO_FLUSH
|Flush (write-back + invalidate) specific cache line matching address

|2'b10
|CMO_CLEAN
|Write back dirty line without invalidating (write-back mode only)

|2'b11
|CMO_FLUSH_ALL
|Write back all dirty lines; invalidate all entries (DMA pre/post-transfer use)
|===

NOTE: `CMO_FLUSH_ALL` is triggered by `FENCE.I` and is required before DMA transfers to maintain coherence.

=== Performance Counters (Simulation Only)

The cache provides performance monitoring signals (not synthesized):

* `perf_req_cnt`: Total data accesses
* `perf_hit_cnt`: Cache hits (load or store)
* `perf_miss_cnt`: Cache misses
* `perf_bypass_cnt`: Non-cacheable bypasses (PMA or CMO_DISABLE)
* `perf_fill_cnt`: Completed line fills
* `perf_evict_cnt`: Dirty line evictions
* `perf_cmo_cnt`: CMO operations executed

==== Transfer Modes

===== 1D Transfer (Linear)

Simple contiguous memory copy:

* Set `SRC_ADDR`, `DST_ADDR`, `XFER_CNT`
* Set `CTRL[4:3]=00` (MODE=1D)
* Set `CTRL[5:6]` for address increment behavior
* Set `CTRL[1]=1` to start

===== 2D Transfer (Strided)

Copy 2D arrays with row stride:

* Set `SRC_ADDR`, `DST_ADDR`, `XFER_CNT` (bytes per row)
* Set `SRC_STRIDE`, `DST_STRIDE` (bytes between row starts)
* Set `ROW_CNT` (number of rows)
* Set `CTRL[4:3]=01` (MODE=2D)

Example: Copy 16×16 pixel subregion from 640×480 framebuffer:

----
XFER_CNT = 16 * 4        // 16 pixels × 4 bytes/pixel
SRC_STRIDE = 640 * 4     // Source pitch
DST_STRIDE = 16 * 4      // Destination pitch (packed)
ROW_CNT = 16             // 16 rows
----

===== 3D Transfer (Planar)

Copy 3D volumes with plane stride:

* Configure as 2D, plus set `SRC_PLANE_STRIDE`, `DST_PLANE_STRIDE`, `PLANE_CNT`
* Set `CTRL[4:3]=10` (MODE=3D)

===== Scatter-Gather Transfer

Chain multiple transfers from descriptor list:

* Each descriptor is 16 bytes: `{src_addr, dst_addr, xfer_cnt, mode_ctrl}`
* Set `SG_ADDR` to descriptor list base address
* Set `SG_CNT` to number of descriptors
* Set `CTRL[4:3]=11` (MODE=SG)

==== Interrupt Behavior

* Each channel can raise interrupt on completion (DONE) or error (ERR)
* Channel interrupts are aggregated to single PLIC source (ID 4)
* Software reads `IRQ_STAT` to determine which channel(s) triggered
* Write 1 to `IRQ_STAT` bits to acknowledge

==== Performance Monitoring

Enable performance counters to measure DMA throughput:

----
DMA_PERF_CTRL = 0x3;    // Enable counters and reset
// ... run workload ...
cycles = DMA_PERF_CYCLES;
rd_bytes = DMA_PERF_RD_BYTES;
wr_bytes = DMA_PERF_WR_BYTES;
bandwidth_MBps = (rd_bytes + wr_bytes) / cycles * CLK_FREQ / 1e6;
----

==== AXI Interface

The DMA controller has a dedicated AXI4 master port (32-bit data width default):

* Burst optimization: DMA controller generates INCR bursts up to `MAX_BURST_LEN` beats
* Page boundary handling: Bursts never cross 4KB AXI page boundaries
* Error handling: AXI SLVERR/DECERR sets channel STAT[2]=1 and aborts transfer

==== Timer Modes

*Simple mode* — periodic interrupt, period = COMPARE2 + 1 ticks:

. Set COMPARE1 = 0, COMPARE2 = period - 1
. Set CTRL = EN | INT_EN | PRESCALE(n)
. On interrupt: read/clear INT_STATUS; counter auto-reloads

*Dual compare mode* — two events per period:

. COMPARE1 fires a mid-period interrupt (e.g. for phase-correct PWM updates)
. COMPARE2 fires a period-end interrupt and reloads the counter

*PWM mode*:

. Set CTRL[1] = PWM_EN; output goes high at COUNT=0, low at COUNT=COMPARE1, reloads at COMPARE2
. Duty cycle = COMPARE1 / (COMPARE2 + 1)
. Update COMPARE1 on-the-fly to adjust duty without stopping the timer

==== Timing Example (100 MHz clock, 1 kHz interrupt)

----
PRESCALE = 0         // No prescaling; clock = 100 MHz
COMPARE2 = 99999     // Period = 100000 ticks = 1 ms
INT_ENABLE = 0x1     // Enable channel 0 global IRQ
CTRL = EN | INT_EN   // Start timer with interrupt
----

For a 1 kHz 50% duty-cycle PWM:

----
COMPARE1 = 49999     // Falling edge at 50% duty
COMPARE2 = 99999     // Period = 1 ms
CTRL = EN | PWM_EN | PWM_POL | INT_EN
----

==== Interrupt Behavior

* Each channel's interrupt is enabled by both CTRL[3] (channel) and INT_ENABLE[ch] (global)
* INT_STATUS bit is set on COMPARE1 match (if dual mode) or COMPARE2 match
* Write-1-to-clear INT_STATUS[ch] in ISR; timer continues running
* Each channel has an independent PLIC source: channel N maps to PLIC source `KV_PLIC_SRC_TIMER0 + N` (sources 6–9)

=== Watchdog Timer (WDT)

==== Overview

The WDT provides a hardware safety net that resets the system (or raises an IRQ) if software fails to periodically service it.

[cols="1,3"]
.WDT Features
|===
|Feature |Specification

|Base Address
|0x2006_0000

|Size
|64KB

|PLIC Source
|10 (`KV_PLIC_SRC_WDT`) — only when INTR_EN=1

|External output
|`wdt_reset_o` — asserts and latches when INTR_EN=0 and expiry occurs; cleared only by `rst_n`

|Counter width
|32-bit countdown (configurable via LOAD register)
|===

==== Register Map

[cols="1,1,1,4"]
.WDT Register Map
|===
|Offset |Name |Access |Description

|0x00
|CTRL
|R/W
|[0]=EN: start/stop the countdown. +
[1]=INTR_EN: expiry action (1=assert PLIC IRQ, 0=assert `wdt_reset_o`)

|0x04
|LOAD
|R/W
|Reload value — written before enabling the WDT. Value loaded into COUNT on enable or KICK.

|0x08
|COUNT
|RO
|Current countdown value. Counts down from LOAD to 0.

|0x0C
|KICK
|WO
|Any write reloads COUNT from LOAD (\"pets\" the watchdog). Ignores written data.

|0x10
|STATUS
|R/W1C
|[0]=WDT_INT — set when count expires with INTR_EN=1. Write 1 to clear.

|0x14
|CAP
|RO
|Capability: `0x0001_0020` (version=1, counter width=32)
|===

Addresses > 0x17 within the WDT aperture return SLVERR.

==== Operation

1. Write reload value to LOAD.
2. Write `0x03` to CTRL (EN=1, INTR_EN=1 for IRQ mode, or `0x01` for reset mode).
3. Periodically write any value to KICK before COUNT reaches 0.

On expiry (COUNT reaches 0)::
* **IRQ mode** (INTR_EN=1): STATUS[0] is set; PLIC source 10 asserts. Software must claim, handle, kick (or clear STATUS), and complete.
* **Reset mode** (INTR_EN=0): `wdt_reset_o` asserts and _latches_ until `rst_n`; software cannot recover. System integrator feeds `wdt_reset_o` into the system reset tree.

The COUNT register latches at zero after expiry; a KICK is required to resume counting.

==== Interrupt Behavior

* `wdt_irq` to PLIC = `STATUS[0] & CTRL[1]` (only fires in IRQ mode)
* `wdt_reset_o` = latched register set on expiry in reset mode; cleared only by `rst_n`
* In simulation (`kv32sim`): reset mode calls `exit(2)`; Spike plugin calls `sim->exit(2)`

==== External Pins

[cols="1,1,4"]
.WDT External Pins
|===
|Pin |Direction |Description

|wdt_reset_o
|Output
|Hardware reset request. Latches high on reset-mode expiry. System integrator feeds into reset tree.
|===

----

Typical use: Inter-processor interrupt (IPI) on multi-hart systems, or self-interrupt for context switching.

==== Interrupt Flow

1. **Pending**: Peripheral asserts `irq_src[i]`, PLIC sets `PENDING[i]=1`
2. **Arbitration**: PLIC selects highest-priority enabled pending interrupt where `priority[i] > threshold`
3. **Notification**: PLIC asserts `external_irq` to core
4. **Claim**: ISR reads `CLAIM` register to get source ID, `PENDING[i]` remains set
5. **Service**: ISR handles the interrupt, clears condition in peripheral
6. **Complete**: ISR writes source ID to `COMPLETE` register
7. **Clear**: If peripheral de-asserted `irq_src[i]`, PLIC clears `PENDING[i]`

NOTE: If peripheral IRQ remains asserted after complete, `PENDING[i]` is immediately re-set (level-sensitive behavior).

==== Priority Arbitration

When multiple interrupts are pending:

* Highest priority wins (priority 7 > priority 1)
* On priority tie, lowest source ID wins (source 1 > source 2)

Example configuration:

----
PRIORITY[1] = 5;  // UART high priority
PRIORITY[2] = 3;  // SPI medium priority
PRIORITY[3] = 3;  // I2C medium priority
PRIORITY[4] = 1;  // DMA low priority
PRIORITY[5] = 4;  // GPIO medium-high priority
PRIORITY[6] = 2;  // Timer low-medium priority
THRESHOLD = 0;    // Allow all enabled interrupts
ENABLE = 0x7E;    // Enable sources 1-6
----

=== Crossbar Architecture

The AXI crossbar (`axi_xbar`) routes transactions from the core to appropriate slave devices:

* **Master Port**: Connected to memory arbiter (combines I-fetch + D-access paths)
* **Slave Ports**: 10 devices (RAM, CLINT, PLIC, UART, I2C, SPI, DMA, GPIO, Timer, Magic)

==== Address Decode Logic

Decoding is priority-based (higher-numbered slaves checked first):

[source,systemverilog]
----
if (addr[31:16] == 16'hFFFF)       → Slave 9 (Magic)
else if (addr[31:16] == 16'h2005)  → Slave 8 (Timer)
else if (addr[31:16] == 16'h2004)  → Slave 7 (GPIO)
else if (addr[31:12] == 20'h2003_0) → Slave 6 (DMA)
else if (addr[31:16] == 16'h2002)  → Slave 5 (SPI)
else if (addr[31:16] == 16'h2001)  → Slave 4 (I2C)
else if (addr[31:16] == 16'h2000)  → Slave 3 (UART)
else if (addr[31:26] == 6'h03)     → Slave 2 (PLIC)
else if (addr[31:20] == 12'h020)   → Slave 1 (CLINT)
else if (addr[31:21] == 11'h400)   → Slave 0 (RAM)
else                               → SLVERR (unmapped)
----

==== Error Responses

* **Unmapped address read**: Crossbar returns `SLVERR` response with data = 0xDEADBEEF
* **Unmapped address write**: Crossbar returns `OKAY` response (write silently dropped)

==== I-Cache Burst Transactions

The instruction cache uses WRAP bursts for critical-word-first cache-line fills:

* Burst type: `ARBURST = 2'b10` (WRAP)
* Burst length: `ARLEN = (line_size / 4) - 1` (e.g., 15 for 64-byte line)
* Starting address: Critical word address (PC)
* Wrap boundary: Cache-line boundary

==== D-Cache Burst Transactions

The data cache uses WRAP bursts for cache-line fills and INCR for dirty evictions and write-through:

* **Fill (read)**: `ARBURST = 2'b10` (WRAP), `ARLEN = (line_size / 4) - 1` (e.g., 7 for 32-byte line)
* **Dirty eviction (write)**: `AWBURST = 2'b01` (INCR), `AWLEN = WORDS_PER_LINE - 1`, full cache-line burst starting at line base address
* **Write-through (write)**: `AWBURST = 2'b01` (INCR), `AWLEN = 0` (single beat), byte strobes indicate sub-word granularity
* **Non-cacheable bypass**: `ARBURST = 2'b01` (INCR), `ARLEN = 0` (single beat) for reads; same for writes

Example: D-cache WRAP fill from address 0x80000054 with 32-byte cache line:

----
ARCHR = 0x80000054
ARLEN = 7           // 8 beats
ARBURST = WRAP
Beats returned:
  0: 0x80000054 (critical word, returned first)
  1: 0x80000058
  2: 0x8000005C
  3: 0x80000040 (wrap to line boundary)
  4: 0x80000044
  5: 0x80000048
  6: 0x8000004C
  7: 0x80000050
----

Example: I-cache WRAP fill from address 0x80000014 with 64-byte cache line:

----
ARADDR = 0x80000014
ARLEN = 15          // 16 beats
ARBURST = WRAP
Beats returned:
  0: 0x80000014 (critical word, returned first)
  1: 0x80000018
  2: 0x8000001C
  3: 0x80000020
  4: 0x80000024
  5: 0x80000028
  6: 0x8000002C
  7: 0x80000030
  8: 0x80000034
  9: 0x80000038
  A: 0x8000003C
  B: 0x80000000 (wrap to line boundary)
  C: 0x80000004
  D: 0x80000008
  E: 0x8000000C
  F: 0x80000010
----

NOTE: D-Cache also uses WRAP bursts for fills; see Section 6 for the D-Cache-specific example.

==== Data Access Transactions

Data loads/stores without D-cache use single-beat INCR bursts. When D-cache is enabled, most accesses are served from cache without AXI traffic; only misses, dirty evictions, and write-through stores generate AXI transactions (see <<Data Cache Specification>>).

== Programming Guide

=== Boot Sequence

The processor boots from address `0x8000_0000` (RAM base) on reset:

1. Reset vector: PC = 0x8000_0000
2. Bootloader/firmware executes (typically copied to RAM by JTAG or external loader)
3. Initialize stack pointer
4. Initialize CSRs (mstatus, mtvec, mie)
5. Initialize peripherals (UART, timers, etc.)
6. Enable interrupts (clear mstatus.MIE)
7. Jump to main application

=== Memory Initialization

For simulation, the ELF loader (`elfloader.cpp`) loads program sections into RAM via DPI-C before simulation starts. For FPGA, use:

* **JTAG**: Load program via JTAG debugger (OpenOCD, etc.)

=== Interrupt Handling

==== Setting Up Interrupts

[source,c]
----
#include "csr.h"
#include "kv_platform.h"
#include "kv_plic.h"
#include "kv_clint.h"

void setup_interrupts(void) {
    // Set trap vector (must be 4-byte aligned)
    write_csr(mtvec, (uint32_t)trap_entry);

    // Enable M-mode interrupts in mstatus
    set_csr(mstatus, MSTATUS_MIE);

    // Enable specific interrupt types in mie
    set_csr(mie, MIE_MEIE | MIE_MTIE | MIE_MSIE);

    // Configure PLIC
    kv_plic_set_priority(KV_PLIC_SRC_UART, 5);
    kv_plic_enable_interrupt(KV_PLIC_SRC_UART);
    kv_plic_set_threshold(0);

    // Configure CLINT timer for periodic interrupt (1ms)
    uint64_t mtime = kv_clint_read_mtime();
    kv_clint_write_mtimecmp(mtime + 100000);  // +1ms @ 100MHz
}

==== Trap Handler Example

[source,c]
----
void trap_handler(void) {
    uint32_t mcause = read_csr(mcause);
    uint32_t mepc = read_csr(mepc);

    if (mcause & 0x80000000) {
        // Interrupt
        uint32_t irq_code = mcause & 0x1F;
        switch (irq_code) {
        case 7:  // Timer interrupt
            handle_timer_interrupt();
            break;
        case 11: // External interrupt (PLIC)
            handle_external_interrupt();
            break;
        case 3:  // Software interrupt
            handle_software_interrupt();
            break;
        }
    } else {
        // Exception
        handle_exception(mcause, mepc);
    }
}

void handle_timer_interrupt(void) {
    // Advance mtimecmp for next interrupt
    uint64_t mtimecmp = kv_clint_read_mtimecmp();
    kv_clint_write_mtimecmp(mtimecmp + 100000);  // +1ms
}

void handle_external_interrupt(void) {
    // Claim interrupt from PLIC
    uint32_t source = kv_plic_claim_interrupt();

    switch (source) {
    case KV_PLIC_SRC_UART:
        uart_isr();
        break;
    case KV_PLIC_SRC_SPI:
        spi_isr();
        break;
    // ... handle other sources ...
    }

    // Complete interrupt
    kv_plic_complete_interrupt(source);
}
----

==== SPI Flash Read

[source,c]
----
#include "kv_spi.h"

#define SPI_CMD_READ 0x03

void spi_read_flash(uint32_t addr, uint8_t *buf, size_t len) {
    kv_spi_init(1000000, KV_SPI_MODE0);  // 1 MHz, Mode 0

    kv_spi_select(0);  // Assert CS[0]

    // Send READ command + 24-bit address
    kv_spi_transfer(SPI_CMD_READ);
    kv_spi_transfer((addr >> 16) & 0xFF);
    kv_spi_transfer((addr >> 8) & 0xFF);
    kv_spi_transfer(addr & 0xFF);

    // Read data
    for (size_t i = 0; i < len; i++) {
        buf[i] = kv_spi_transfer(0x00);
    }

    kv_spi_deselect(0);  // De-assert CS[0]
}

[source,c]

#include "kv_i2c.h"

#define EEPROM_ADDR 0x50  // 7-bit I2C address

void i2c_eeprom_write(uint8_t mem_addr, uint8_t data) {
    kv_i2c_init(100000);  // 100 kHz

    kv_i2c_start();

=== Performance Optimization Tips

1. **Enable I-Cache**: Set `ICACHE_EN=1` parameter for significant speedup (~2-5× depending on workload)
2. **Enable D-Cache**: Set `DCACHE_EN=1` for data-access speedup on load/store-intensive workloads (~1-cycle load latency on hit vs. 3+ cycles to RAM)
3. **Tune Cache Write Policy**: Use `DCACHE_WRITE_BACK=1` (default) for best performance; use `DCACHE_WRITE_BACK=0` (write-through) where strong write ordering is required
4. **Increase Buffer Depths**: Larger `IB_DEPTH` and `SB_DEPTH` improve throughput in memory-intensive code
5. **Use DMA for Bulk Transfers**: Offload large memory copies to DMA controller; issue `cbo.flush` / `FENCE.I` before/after DMA for coherence
6. **Minimize Load-Use Dependencies**: Reorder instructions to avoid pipeline stalls
7. **Use Fast Multiply/Divide**: Set `FAST_MUL=1` and `FAST_DIV=1` for single-cycle operations
8. **Align Data Structures**: 4-byte alignment avoids misaligned access penalties
9. **Batch Peripheral I/O**: Use FIFOs to buffer multiple bytes before checking status
